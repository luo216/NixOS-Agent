# 智能调试

## 设计目标

系统中的调试能力并不局限于一次性执行 shell 指令，而是强调对复杂运行过程的持续监督、交互式控制和结果验证。尤其在 NixOS 配置、远程检查和长时间命令场景下，传统单次命令执行方式缺乏足够的灵活性和可观察性，因此需要更适合真实工程场景的智能调试机制。

## tmux Skill 的定位

`tmux-session-runner` 是系统智能调试能力的关键支撑。它为 `worker` 提供独立的 tmux 会话和 TTY 环境，使系统能够在一个持续存在的 shell 中执行命令、轮询状态、查看输出并根据需要继续操作，而不是把执行限制在单次命令调用中。

## 长命令监督

对于 `nix build`、`nixos-rebuild`、`home-manager switch`、远程日志观察等长时间任务，系统会优先交给 `worker` 在 tmux 会话中执行。执行后，系统可以持续检查会话状态和最近输出，从而形成一种“边运行、边监督、边反馈”的调试方式。

## 交互式 CLI 支持

由于 tmux 会话提供了独立的终端环境，系统不仅能执行普通命令，还可以进入需要交互的命令行程序或连续调试流程。这种能力使系统更接近真实人工排查过程，也显著提升了复杂调试场景中的灵活性。

## SSH 验证能力

智能调试机制还支持在 tmux 会话中进入 SSH 交互环境，并在同一远端 shell 内连续执行验证、排查与观察命令。与传统“一次发一条远程命令”的方式相比，这种方法更适合验证配置是否真正生效，也更适合处理需要上下文连续性的远程检查任务。

## 机制价值

基于 tmux skill 的智能调试机制，使系统具备了长命令监督、交互式终端操作和远程持续验证三类能力，从而突破了传统智能体只会发起一次性命令的局限。
