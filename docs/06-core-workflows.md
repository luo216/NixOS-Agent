# 核心工作流

## 主流程概览

系统的典型工作流并不是简单的“用户提问，智能体回答”，而是一个由规划、实现、监督、验证和记录组成的闭环过程。这个过程体现了多角色协作、长时运行智能体、智能调试和运行状态可视化之间的协同关系。

## 从需求到实现的流程

在一个完整任务中，`planner` 首先负责理解需求并整理任务拆分，然后将明确的执行目标交接给 `implementer`。`implementer` 根据任务状态逐项完成实现，当遇到需要长时间运行、持续观察、交互式调试或远程验证的场景时，再将子任务交给 `worker`。`worker` 完成监督和验证后，把结果回传给 `implementer` 继续整合与收尾。

## 从执行到可视化的流程

执行过程中，底层运行时会持续产生输出与状态变化。后端负责对这些信息进行整理，并通过 WebSocket 推送给前端，前端再将其显示为时间线、进度面板和 git 变化视图。这样，系统形成了从任务执行到状态展示的完整链路。

## 从结果到沉淀的流程

一个功能完成后，系统不仅要保留最终代码改动，还要通过 `planned-features.json` 和 `codex-progress.md` 记录任务状态与执行过程。这样既能支撑后续任务延续，也能为实验分析和结果复盘提供依据。

## 工作流意义

核心工作流体现了本系统与普通问答系统的根本差异，即系统关注的不只是回答本身，而是任务如何被分解、执行、监督、验证和记录。
