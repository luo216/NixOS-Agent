# 多角色协作

## 设计动机

复杂开发任务往往同时包含需求理解、方案规划、代码实现、验证监督与结果回收等多个阶段。如果全部交由单一智能体完成，容易出现职责混杂、上下文污染和阶段切换失控的问题。因此，系统采用多角色协作模式，将不同类型的任务分配给更明确的执行主体。

## 角色划分

系统当前固定包含 `planner`、`implementer` 与 `worker` 三个角色。

- `planner` 负责需求澄清、任务拆解与方案组织
- `implementer` 负责具体实现、结果整合与交付闭环
- `worker` 负责长命令监督、运行验证与辅助执行

这种划分使系统能够把“想清楚怎么做”“真正改动内容”“持续观察执行结果”分离开来。

## 协作方式

系统中的多角色协作不是自由混聊，而是基于明确交接链路进行。当前主要交接方向为 `planner -> implementer -> worker -> implementer`。交接过程通过 activation 机制显式建模，从而避免人工复制上下文导致的信息偏差，也使协作关系可以被追踪和展示。

## 知识库与技能包支撑

为了提升不同角色在特定任务中的专业性，系统引入了知识库与技能包体系。知识库用于提供 NixOS、nixpkgs 与工程实践相关的专业参考，技能包则用于约束和增强角色的执行能力。它们共同决定了角色“知道什么”和“会怎么做”，因此属于多角色协作能力的一部分，而不是独立于协作体系之外的附属模块。

## 协作收益

多角色协作带来的主要收益包括：角色职责更清晰、长上下文污染更少、任务交接过程更稳定、专业知识调用更有针对性，以及执行监督可以与实现工作并行展开。
